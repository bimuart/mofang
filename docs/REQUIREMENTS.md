## 2. 背景与问题陈述

### 2.1 要解决什么问题

用户希望在**纯静态 HTML + JavaScript** 环境中，完成一个 **3 阶（3×3×3）魔方** 的：

1. **状态定义**：用颜色（或等价地，六个面的 **面记号 U、D、L、R、F、B**）描述当前魔方上每一块贴纸的归属；
2. **交互式编辑**：在符合魔方几何的前提下，通过 **3D 面上点选**（详见 §2.4）为 26 个「单元块」指定或推导颜色；
3. **路径输出**：在状态完全确定且**整体可解**时，给出**还原步骤**（从当前态到某一约定还原态），以及**逆向步骤**（从还原态回到用户定义的态）；含 **Kociemba** 求解、**可解性**校验与 **约束推理**；
4. **三维展示与步骤演示**：以 **3D** 呈现魔方；**面转动**仅由 **步骤演示**（或撤销演示步）触发，**禁止用户自由拧魔方**（详见 §2.4）。展示步骤时 **高亮当前步骤** 并绘制 **箭头**，**手动执行**当前步后**进入下一步**。用户 **可随时调整观察视角**（§2.4）。

### 2.2 为什么是「静态页 + JS」

不依赖后端服务，便于离线部署与分享；所有逻辑在浏览器内完成。复杂求解（如两阶段法）可以纯前端实现，但需注意性能与代码体积，这属于非功能需求范畴。

**与 §4 的关系**：源码采用 **Vite + TypeScript + Vue** 等现代前端工具链；**交付与运行**为构建产物（`dist/`）下的 **静态 HTML / JS / 资源**。**部署与 `file://` 限制**见 **§5 NFR-1**。

### 2.3 核心名词（简表）

更严格的定义见 [几何约束文档](./GEOMETRIC_CONSTRAINTS.md)。

| 名词 | 含义 |
|------|------|
| **面 / 面记号** | 魔方六个外表面；本项目采用 **Singmaster 惯例**：**U、D、L、R、F、B**（分别为 Up / Down / Left / Right / Front / Back，即 **上、下、左、右、前、后**），与转动记号一致；每个面一种颜色，与中心块一一对应。 |
| **中心块** | 每个面中心、只暴露一种颜色的块；共 6 个，决定「每个面对应哪种颜色」。 |
| **棱块** | 暴露两种颜色的块；共 12 个。 |
| **角块** | 暴露三种颜色的块；共 8 个。 |
| **单元块 / 块** | 上述 26 个可移动小块的统称（不含内部不可见的核心结构）。 |
| **还原态** | **恒等于**：用户设定的 **六个中心颜色** 所确定的 **每面整面同色**（U、D、L、R、F、B 各一色，与该面中心块颜色一致）。无其它还原目标。 |
| **定义态** | 用户通过界面配置完成后的魔方状态。 |
| **演示模式** | 已生成还原/逆向步骤列表后，按序 **执行步骤动画** 的流程；此间仍 **禁止自由面拧**（§2.4）。 |
| **转视角** | 调整 **相机/观察**（轨道旋转、缩放、平移等），**不改变**魔方逻辑状态；**任意阶段必须允许**（§2.4）。 |
| **自由拧 / 面转动交互** | 用户通过拖拽、手势等触发 **Singmaster 意义下的面转动**；**全文禁止**（§2.4）；与转视角区分。 |
| **当前步骤** | 步骤序列中正在指示/待执行的那一条；用户 **手动** 执行该步后索引递增（见 FR-20）。 |

### 2.4 已确认的产品与交互约定（实现须遵守）

以下已由产品/需求方确认，开发 **直接按此实现**；若变更须更新本文档版本。

| 项 | 约定 |
|----|------|
| **功能范围** | **不裁剪**：**填色**、**约束推理**（FR-11～FR-13）、**可解性**（FR-6～FR-9）、**Kociemba** 求解、**3D** 展示、**步骤驱动的拧动动画**、**步骤箭头**与列表高亮、**演示同步**（FR-16～FR-22）均在交付范围内。 |
| **填色入口** | 用户在 **3D 魔方** 上 **点选** 指定颜色；**粒度仅为单个贴纸（单格）**，不提供整面一键填充作为唯一方式。不得仅依赖 2D 展开图作为唯一入口。 |
| **还原态定义** | **永远**为：**用户设定的六中心颜色** + **六个外表面各为整面纯色** 且与对应中心颜色一致（见名词「还原态」）。 |
| **步骤推进（FR-20）** | **禁止**「动画结束后自动进入下一步」；**必须**由用户 **手动** 触发「执行当前步 / 下一步」，避免漏看步骤。 |
| **空间坐标（俯视）** | 自 **上向下看**（沿 **U** 朝下俯视），四个侧面水平环绕顺序 **顺时针** 为 **F → L → B → R**（闭环回到 F）。3D 模型中各面 mesh/法线与 Singmaster 记号、步骤箭头须与此一致；**Singmaster 单面转动的「顺时针」仍定义为面向该面正视时的顺时针**（与几何文档一致）。 |
| **约束推理（FR-11）** | 用户每次填色或变更后，须 **实时** 重算各未决贴纸的 **候选颜色** 并刷新 UI，不得仅依赖「点击应用后再推理」。 |
| **转视角（必须）** | **所有界面状态**（填色、空闲、演示步骤、求解结果展示等）下，均须允许用户 **调整观察视角**（轨道旋转、缩放、平移或等价操作），以便点选各面贴纸、看清箭头与高亮。**视角变换不得**改变魔方 **逻辑状态**（仅改变相机/观察）。 |
| **禁止自由拧魔方（全局）** | **任何时刻** 禁止用户通过拖拽、滚动手势等对魔方施加 **面转动**（即 **Singmaster 意义下的 U/D/L/R/F/B 层转动**）。魔方 **逻辑状态** 仅允许通过：**填色**（FR-10）、**还原/逆向步骤演示**（FR-17/20）、以及 **若实现 FR-22** 时的 **撤销演示步** 改变。不得提供「练习拧魔方」「鼠标拖面旋转」等交互。 |
| **演示模式与 FR-21** | 播放 **还原/逆向步骤** 时进入 **演示模式**；除上述 **步骤执行 / 撤销** 外，仍 **不得** 自由面拧，与全局规则一致。 |

**实现层须在 README 中写清**：Kociemba **54 字符 facelet 串** 与贴纸顺序（**§4**）；**部署方式**（**§5 NFR-1**）；**相机与禁止面拧**（§2.4、FR-16/21、**§3.6** 末实现提示）。

## 3. 功能需求

### 3.1 面与颜色（面记号 U、D、L、R、F、B）

- **FR-1**：系统应使用 **六个面记号 U、D、L、R、F、B** 表示六个物理面（与 **Singmaster / 标准魔方记号** 一致）；每个记号对应一种颜色（展示层可用色块或色名，内部以面记号或固定枚举为准亦可）。
- **FR-2**：用户应能 **确认六面初始颜色**，即 **6 个中心块** 各自所在面（U、D、L、R、F、B）上的颜色。该中心配置作为后续「还原态」的基准。

### 3.2 26 个单元块与相邻色

- **FR-3**：系统应能区分并处理 **26 个单元块**：6 中心 + 12 棱 + 8 角。
- **FR-4**：对每一种块类型，应能描述 **各外露面上的颜色（或所属面记号）**，即「该块在空间中每一可见贴纸位置」与颜色的对应关系；其中：
  - 中心块：1 面；
  - 棱块：2 面；
  - 角块：3 面。
- **FR-5**：实现应遵循 **空间相邻关系**：仅当两个贴纸在真实魔方上共边相邻时，才允许建立「相邻色」关系；不得出现物理上不相邻的面被错误关联。

### 3.3 几何与可解性约束（产品层）

以下与数学细节对应的条文，在 [GEOMETRIC_CONSTRAINTS.md](./GEOMETRIC_CONSTRAINTS.md) 中有定义与检验说明。

- **FR-6（色位 / 块身份）**：任意合法状态下，每个棱块仅携带两种颜色、每个角块仅携带三种颜色；**不可能**通过旋转得到「与块类型不符」的贴纸组合（例如角块上出现两种相同颜色冒充第三种）。
- **FR-7（角块朝向守恒 — 全局）**：在一致的角块朝向定义下，**8 个角块扭转量之和 ≡ 0 (mod 3)**。
- **FR-8（棱块翻转 — 全局）**：在一致的棱块朝向定义下，**处于「已翻转」状态的棱块个数为偶数**。
- **FR-9（排列奇偶 — 全局）**：相对约定还原态，**角块位置置换与棱块位置置换的奇偶性必须相同**（详见几何文档）；仅检查角扭转与棱翻转 **不足以** 保证可解，**必须**实现或等价地检验本条件。

### 3.4 用户自定义位置与自动推理

- **FR-10**：用户应能通过 **在 3D 模型上点选单个贴纸（单格）** 并 **颜色选择**，为各单元块的各外露面指定颜色（或面记号）；未指定的贴纸可保持「未知」。
- **FR-11**：对于未确定的贴纸，系统应根据 **已确定信息 + FR-5～FR-9** 给出 **仍允许的颜色候选**；候选须 **实时** 随用户填色/变更更新（见 §2.4）。
- **FR-12**：当某一贴纸的候选集合 **缩减为唯一颜色** 时，系统应 **自动赋值**（或明确提示并赋值，行为需在 UI 中一致），并继续推理直至所有贴纸确定或进入用户必须手动选择的分支。
- **FR-13**：若当前部分赋值无法延拓为任何满足 FR-6～FR-9 的全局赋值，系统应能 **检测冲突** 并提示用户（例如回溯或高亮矛盾处）。

#### 3.4.1 填色扩展：锁定、三级候选、冲突判定与调色板（实现约定）

以下在 **FR-10～FR-13** 之上细化交互与两层「候选」语义，便于实现与验收对齐。

| 概念 | 说明 |
|------|------|
| **格状态** | 每个贴纸格有 **颜色**（含未选）与 **锁定**；六 **中心格** 默认锁定，逻辑色恒为对应面记号 **U～B**（显示色可单独配置）。 |
| **操作流程** | **先** 在 3D 中点选贴纸，**再** 在侧栏选择面记号颜色；手动赋色后该格 **自动锁定**（非中心格可解锁）。 |
| **初始化** | 「初始化」等价于清空非中心贴纸并恢复六中心与默认锁定。 |
| **严格普通候选** | 在 **仅锁定格固定**、其余未锁定视为可重填的前提下，满足 **块 multiset（FR-6 / 约束 A）**、**全局九色计数**，且存在对 **可解性（FR-6～FR-9）** 的延拓（实现中在未知格过多时允许弱近似）。用于 **冲突判定** 与 **推荐/普通** 分级。 |
| **推荐候选** | 在 **所有已着色格（含未锁定）** 均保持的前提下，仍可延拓的颜色。 |
| **冲突格** | 已着色且 **当前色 ∉ 严格普通候选**（未选色不算冲突）。冲突格在 3D 上 **高亮闪烁**；冲突集在 **锁定或赋色变更** 时刷新。 |
| **冲突子集内互斥** | 对处于 **冲突集** 内的格计算 **严格普通候选** 时，冲突格之间 **不互为已着色约束**，非冲突格保持当前色；见实现 `buildPartialNormalInConflict`。 |
| **调色板可点（宽松）** | 仅校验 **锁定格固定** + **全局每种颜色至多 9 枚、未决格数匹配**（`globalCountFeasible`），**不**单独用块 multiset 限制单格。这样在 **仅六中心已填** 的空白态下，任一角/棱格均可点选 **六种** 面记号，避免未完成填色时误把角块限制为 3 色、棱块为 2 色而导致「无法点选」。 |
| **调色板三级样式** | **推荐**：严格几何下为推荐候选；**普通**：严格普通候选但非推荐；**禁止样式**：严格几何下非普通候选（仍可能可点，若宽松调色板允许），用于提示「可能产生冲突」。 |
| **求解** | 存在 **冲突格** 时禁止启动「还原 / 逆向」；完全填色且 **无冲突** 且可解性通过后方可求解。 |

### 3.5 还原与逆向步骤

- **FR-14**：当 **所有单元块位置与颜色均确定**，且整体状态满足 **可解性**（FR-6～FR-9），用户可点击 **「还原」**，系统输出从 **当前定义态 → 还原态** 的转动步骤序列（步骤格式需在实现中固定，如 Singmaster 记号：U、D、L、R、F、B 及 `'`、`2` 等）。
- **FR-15**：用户可点击 **「逆向」**，系统输出从 **还原态 → 当前定义态** 的步骤序列（应为 FR-14 序列的逆，或等价算法生成；需与内部表示一致）。

### 3.6 三维展示、拧动模拟与步骤同步

本节约定 **3D 视图**、**步骤列表** 与 **魔方真实状态** 的一致性与交互细节，避免「列表与模型脱节」。

- **FR-16（3D 呈现与观察）**：魔方应以 **三维** 形式展示，用户能辨认六个面及贴纸颜色（或与 **U～B 面记号** 对应的色），空间关系符合 3×3 几何。**须始终提供转视角**（见 §2.4）。**禁止**提供使用户得以 **自由面拧** 的交互（与 FR-21 一致）。
- **FR-17（拧动模拟 — 仅步骤驱动）**：系统应能按与 **FR-14 / FR-15** 一致的记号（如 U、D、L、R、F、B 及 `'`、`2`）对魔方做 **旋转动画**，且动画结束后 **内部状态** 与该步语义一致。**面转动动画**仅允许由 **步骤演示流程**（FR-20）或 **撤销演示步**（若实现 FR-22）触发，**不得**由用户拖拽魔方面触发。
- **FR-18（单一状态源）**：**步骤列表、当前步索引、3D 魔方状态** 应维护为 **同一套逻辑状态的可视化**（或严格可推导）；禁止出现「列表已进到第 k 步而模型仍停留在第 k−1 步之后未更新」且无提示的长期不一致。
- **FR-19（当前步高亮与箭头）**：当存在由 FR-14 或 FR-15 生成的 **非空步骤序列** 且处于 **逐步演示** 流程中时，对 **当前步骤** 应同时满足：
  - **列表侧**：在步骤列表中 **高亮当前行**（与已完成、未执行行视觉区分）；
  - **模型侧**：在 3D 魔方上给出 **该步对应的视觉提示**，至少包括：
    - **箭头**：指示本步的 **旋转轴与方向**（顺/逆时针），含义与 FR-14 记号一致；允许辅以 **文字标签**（如 `R`）消除歧义；
    - **高亮**：对本步涉及的 **面或层** 做 **描边、半透明罩层或变色**，与箭头指向一致。
- **FR-20（执行一步后进入下一步 — 仅手动）**：用户通过 **「执行当前步」/「下一步」**（或等价控件）**手动** 触发后，系统应完成 FR-17 的动画，更新魔方状态，将 **当前步骤索引** 移至 **下一步**；最后一步执行完毕后进入 **结束态**（如禁用「下一步」、或提示序列已完成）。  
  - **禁止**：**不得**提供「动画结束后 **自动** 进入下一步」或自动连播（除非未来需求变更并修订本文档）；避免用户漏看步骤。
- **FR-21（全局禁止自由面拧）**：**任何阶段**（含填色、非演示、演示步骤）均 **禁止** 用户通过交互对魔方进行 **自由面拧**（§2.4）。**演示模式**下除 **执行当前步 / 上一步（FR-22）** 外，同样不得面拧。**转视角** 始终允许且与面拧区分（§2.4）。
- **FR-22（撤销与回退 — 建议）**：在步骤演示流程中，提供 **「上一步」** 或 **撤销最近一次演示转动**，使模型与 **当前步骤索引** 同步回退，便于纠错与复盘。

**实现提示（Three.js，与 §2.4 一致）**：**转视角** 使用轨道控制器（如 `OrbitControls`）或等价物，只动 **相机**（及可选目标点），**不动**魔方组上表示 **层转动** 的变换。**禁止面拧**：勿将「拖动某面」绑定为层旋转；用户输入仅 **点击贴纸**（填色）与 **按钮**（下一步 / 上一步）。若整体旋转魔方根节点仅作视觉辅助，须保证 **不改变** 逻辑状态；**推荐仅用相机** 观察，避免与面拧混淆。

## 4. 技术栈（约定）

以下为 **默认/推荐实现选型**，便于多人协作与 AI Agent 一致实现；若需替换，应在 PR 中说明理由并同步更新本文档。

| 层级 | 选型 | 说明 |
|------|------|------|
| 构建与开发 | **Vite** | 本地开发、打包为静态资源；输出目录一般为 `dist/`。 |
| 语言 | **TypeScript** | 魔方状态、群运算与求解逻辑强类型，减少索引与枚举错误。 |
| UI 框架 | **Vue 3**（**Composition API** + **SFC**） | 步骤列表、填色表单、当前步索引等与 3D 视图 **响应式绑定**；组件边界清晰。 |
| 三维 | **Three.js** | 3×3 几何体、面旋转动画、箭头/高亮可与场景图节点绑定。 |
| 状态（可选） | **Pinia** | 若跨组件共享复杂（定义态、演示序列、3D ref）时使用；否则可用 **composable + provide/inject**。 |
| 包管理 | **pnpm**（推荐） | 与 npm/yarn 二选一即可，团队统一即可。 |
| 求解核心 | **Kociemba 两阶段**（**TypeScript** 实现或经评审的 npm 依赖） | 输出 FR-14/15 所需 Singmaster 序列；具体包名在 `package.json` 与 README 中固定；**输入接口见下段「facelet 串」**。 |
| 代码质量（建议） | **ESLint** + **Prettier** | 与 Vue/TS 官方推荐配置对齐。 |

**Kociemba 输入（54 字符 facelet 串）**：两阶段法等求解器通常接收 **长度 54** 的字符串，与 **6 个面 × 每面 9 个贴纸** 一一对应；每个字符表示该格贴纸属于 **六种颜色中的哪一种**（常用 `U/D/L/R/F/B` 表示「与还原态哪一面中心同色」）。实现时须将应用内部的 54 个贴纸颜色按 **所选 npm 库规定的遍历顺序**（例如各面 9 格的先后，**以库文档为准**）拼成一串传入；**顺序与库不一致则求解错误**。README 中须写死「本项目 facelet 顺序与某某库一致」。此条为 **库接口约定**，与能否静态部署无关。

**选型理由（摘要）**：Vue 与步骤/表单状态天然契合；Three.js 生态成熟、示例多；Vite 开发与构建体验好；TS 对魔方代数与不变量检验更友好。未选 React 仅为减少并列范式、非排斥。

**不采用**：后端运行时、WebAssembly 求解（除非性能证明需要）、Babylon.js（与 Three.js 二选一即可，本文档以 Three 为准）。

## 5. 非功能需求（建议）

- **NFR-1（静态部署与访问方式）**：
  - **正常交付**：`pnpm build` 生成 **`dist/`**（含 `index.html`、带 hash 的 JS/CSS、资源）。将 `dist/` 置于任意 **静态 HTTP 服务**（如 `vite preview`、Nginx、GitHub Pages、`python -m http.server`）即可，**无需后端**，即典型静态站点。
  - **`file://` 与「双击单个 html」**：默认 Vite 等构建使用 **ES 模块** 与模块路径，浏览器在 `file://` 下常 **加载失败白屏**；**不推荐**用资源管理器双击 `index.html` 代替 HTTP。**若必须离线单文件打开**，需额外工程配置（内联、legacy、`base: './'` 等）；默认以 **静态目录 + HTTP** 为准。
  - README 中说明推荐访问方式及 `file://` 限制。
- **NFR-2**：对用户输入做基本校验；在不可解或未完成赋值时，「还原」「逆向」应禁用或明确报错原因。
- **NFR-3**：文档与代码中应写明：**还原态**（§2.4：六中心 + 每面纯色）、**俯视坐标环 F→L→B→R**（§2.4）、**全局禁止自由面拧 + 始终允许转视角**（§2.4）及步骤记号约定，避免歧义。
- **NFR-4**：三维与 UI 实现遵循 **§4 技术栈**；替换 Three.js / Vue 等须经文档同步更新。

## 6. 验收标准（摘要）

| 编号 | 验收项 |
|------|--------|
| AC-1 | 可配置 6 个中心颜色（六个面记号 U、D、L、R、F、B），并作为还原基准。 |
| AC-2 | 可通过 **3D 单格点选** 展示/编辑 26 块及其贴纸，相邻关系符合 3×3 几何；**可转视角**且 **无自由面拧**（§2.4）。 |
| AC-3 | 填色推理符合 [几何约束文档](./GEOMETRIC_CONSTRAINTS.md) 中的四条全局约束（含 **排列奇偶**）。 |
| AC-4 | 唯一候选时自动填色；矛盾时可检测并提示；候选随输入 **实时** 更新（§2.4、FR-11）。 |
| AC-5 | 完全确定且可解时，「还原」「逆向」可生成步骤序列。 |
| AC-6 | **仅**在步骤演示中按记号播放拧动动画，动画结束后状态与单步语义一致（FR-17）；用户 **不能** 自由面拧；可转视角（§2.4、FR-16）。 |
| AC-7 | 演示步骤时，当前步在 **列表** 与 **3D 模型** 上均可识别：列表高亮 + 模型箭头/面高亮（与 FR-19 一致）。 |
| AC-8 | **仅手动**执行当前步后，索引进入下一步；**无**自动连播下一步（与 FR-20 一致）；全程列表与模型无长期脱节（与 FR-18 一致）。 |
| AC-9 | **任意阶段禁止自由面拧**；演示中仅步骤/撤销（若实现 FR-22）改变魔方状态（FR-21）；若实现 FR-22，则撤销后列表与模型一致。 |
| AC-10 | 还原目标为 **六中心 + 每面纯色**（§2.4）；求解为 **Kociemba**，且通过 **可解性 + 约束推理** 与文档一致。 |
| AC-11 | **所有界面状态**下均可 **转视角**；**全程无** 自由面拧交互（§2.4）。 |
